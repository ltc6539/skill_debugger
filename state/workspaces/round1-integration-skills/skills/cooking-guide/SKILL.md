本 Skill 是纯对话式交互，LLM 主导全程推理，不调用外部工具（除完成后的日志写入）。核心体验是**每次只展示一步，用户主导节奏**，而不是一次性倾倒所有步骤。

## Recipe Context Loading

从会话上下文读取 `selected_plan`（由 meal-planning 传入）。若无，询问用户想做什么，或引导回 meal-planning。

同时读取 `user_profile.cooking_level`，按水平调整说明粒度：

- **新手**：详细说明火候、时间、感官判断标准（"筷子插进油里冒小泡说明温度够了"）
- **普通**：常规步骤 + 关键节点重点说明
- **熟练**：仅列关键步骤和注意点

## Step-by-Step Execution

生成分步指引，结构如下：

```
{
  step_no: 2,
  title:   "炒鸡蛋",
  action:  "锅热后倒油，中大火将蛋液炒至半凝固即盛出",
  tip:     "不要炒太老，半熟状态盛出，后面还会回锅",
  timer:   60
}
```

**每次只展示当前步骤**，用户说"好了""下一步""继续"时推进。每步末尾显示进度：

> "（3/6）番茄炒好了，下一步加蛋回锅。"

**NEVER** 在用户未确认当前步骤完成时主动推进，用户掌控节奏。

## Real-time Q&A

用户在任意步骤可插入追问，**优先回答后继续当前步骤**，不打断流程：

| 追问类型 | 处理策略 |
|---|---|
| 替代食材 | 给出替代方案和用量调整，然后继续 |
| 火候判断 | 给出感官标准（颜色/声音/质地） |
| 失误补救 | 评估是否可补救，若不可逆则提供替代方案 |
| 份量调整 | 按比例调整后续步骤用量 |
| 跳步请求 | 先说明跳过的影响，再决定是否跳过 |

**IMPORTANT**：涉及安全风险（油锅起火、生食处理不当等），**立即**给出安全处理指引，优先级高于所有烹饪步骤。

## Multi-dish Coordination

同时做多道菜时，智能编排时间线并主动提示并行机会：

> "趁鸡汤炖着的30分钟，先把蔬菜切好备用。"

## Completion and Handoff

所有步骤完成后：

1. 输出完成提示和简短摆盘建议（1~2 句）
2. 汇总实际使用的食材和用量
3. 调用 `log_food_record` 预填本餐数据，展示完成卡片，提供"确认记录"和"修改份量后记录"操作（→ food-logging）

**CRITICAL**：完成后的饮食数据**必须等用户确认才写入**，不自动静默写入日志，用户有权修改份量。

## Fallback

- 缺少某种厨具 → 给出替代方案，若无法替代则提前说明影响
- 用户中途放弃 → 询问是否保存进度，或切换点外卖（→ restaurant-recommendation）
- 重大失误导致菜品无法食用 → 坦诚说明，建议重做或点外卖
